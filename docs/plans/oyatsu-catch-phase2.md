# おやつキャッチ Phase 2 — 猫・おやつテーマの軽量インクリメンタル

**用語（本ファイル）**

- **VS** は *Vampire Survivors* の略（**レベルアップ時の 2〜3 択**など、**ビルドを選ぶ UI の比喩**として使う）。**「対戦」や Visual Studio の意味ではない。**
- **インクリメンタル**（增量系）: *Cookie Clicker* や放置系に近い、**時間とアップグレードで数値が育つ**遊び。**本作ではポートフォリオ向けに「短いセッション・軽い永続」に収める。**

---

## 0. 位置づけ・方針転換（2026）

| 項目 | 内容 |
|------|------|
| **前身** | Phase 1 は [完了・アーカイブ](done/oyatsu-catch.md)。**現行実装はインクリメンタル（おやつ工房）**に置き換え済み。キャッチ時代の記録は `localStorage` に互換で残る場合あり。 |
| **現在の主軸（確定）** | **猫・おやつ**というテーマに合わせ、**操作をほぼ要らない／放置に近い形**で進み、**やりごたえはビルド・倍率・解放の組み合わせ**から出す。**軽量インクリメンタル**を採用する（参考: *Cookie Clicker*、*TFT* の「監督して選ぶ」感のごく薄い版）。 |
| **やらないこと** | フル画面専用の長期放置ゲーム、本格的なオートバトラー専用クライアント。**トップヒーロー左の枠**に収まる **脇役としての深度**を維持する。 |

**前提技術**: Nuxt / Vue。進行は **`setInterval` / `requestAnimationFrame` / フレームごとの tick** などで足りることが多く、**DOM 中心**でよい。数値・リストが増えても **Canvas は必須ではない**（必要なら部分導入）。

### 0.1 テーマと操作負荷の目標

- **猫**は **感情・ごきげん・報酬の受け手**（右列 `CatMascot` やメーターと接続しやすい）。
- **おやつ**は **生産されるリソース／消費されるアイテム／アップグレードの口実**（数値が育つ理由を一貫させる）。
- **操作**: 基本は **放置**。介入は **アップグレード購入・解放の確定**、**VS 式の 3 択**（レベル到達・イベント・ショップなど）に **集約**するのが望ましい。
- **目安（実装時に調整）**: 1 分あたりの **意図的なタップ／クリックは数回以下**を目標にできる（常時エイムや追従は主軸にしない）。

### 0.2 既存実装（キャッチ）との関係

- 現状の **運搬スプライト移動・落下キャッチ**は、**アクション寄りの別ライン**として実装済み。**Phase 2 のプロダクト方針はインクリメンタル主軸に寄せる**ため、次のいずれかを選ぶ（未決）:
  - **A**: ヒーロー内を **インクリメンタル UI に差し替え**、キャッチは削除または縮小。
  - **B**: **タブ／モード**で「おやつ工房（インクリメンタル）」と「おやつキャッチ」を併存（工数は増える）。
  - **C**: キャッチを **ミニイベント**（稀に発生する短いボーナス）に格下げする。

本ドキュメント以降の **マイルストーン・型設計**は、**A を既定**として書く（差し替え時にファイル名・キー名の整理は別タスク）。

## 1. 借りる「快感の骨格」（インクリメンタル × 選択）

*Vampire Survivors* の **「レベルアップでビルドが変わる」**快感は、**操作ではなく選択**に載せる。*Cookie Clicker* 系の **「倍率が積み上がる」**快感は **生産・コスト曲線**に載せる。

1. **放置でも状況が進む**（ログ・数値が増える／施設が働く）。
2. **短いサイクルで成長が見える**（次の解放までが長すぎないよう、ヒーロー枠向けに圧縮）。
3. **選択の連鎖**（どの強化を先に取るかで、この枠内で「今回はこうなった」が立つ）。
4. **VS 式 3 択**は **レベル・通貨・イベント**などのタイミングで挿入し、**シナジー**を狙わせる。
5. **再プレイ・再訪問**の動機は **未解放・実績・ハイスコア（総生産量など）**に置ける。

## 2. 継承したいもの（サイト・技術）

- **導線**: トップヒーロー左のゲームエリア、`遊ぶ`／閉じる、`AtScroll` 制御（`index.vue`）。
- **永続**: `attt:oyatsu-catch`（またはリネーム後キー）の **単一 JSON + `version`**。
- **`CatMascot`**: リアクション・ごきげん・報酬演出と **数値を結びつけやすい**（操作はさせない）。
- **アクセシビリティ**: `prefers-reduced-motion`、過密アニメを避ける。

---

## 3. コアループ（インクリメンタル）— 設計の柱

### 3.1 リソースと生産

- **おやつ**（または **満腹度／満足度**）など **主リソースを 1〜3 個**に絞る（多すぎると説明が破綻する）。
- **時間経過**（tick）で **生産量が加算**される。施設・猫スキル・バフで **倍率**が掛かる。
- **コスト**で **施設レベル・アップグレード**を購入し、生産を伸ばす（典型的な **コスト曲線**は実装後にチューニング）。

### 3.2 介入（プレイヤーの手）

- **アップグレードの購入**（ボタン数回）。
- **VS 式 2〜3 択**（例: 一定おやつ到達、猫レベル到達、ランダムイベント）。**ラン内専用**でも **メタ解放**でもよいが、最初は **1 本のループ**に収める。
- **プレステージ／転生**は **初期は入れない**。必要になったら `version` 拡張で追加。

### 3.3 猫・おやつで語れる失敗／リスク（任意）

- インクリメンタルは **失敗が薄くなりがち**。**任意**で「悪いおやつ」「機嫌低下」など **軽いデバフ**や **一時的な生産ストップ**を入れられる余地を残す（必須ではない）。

### 3.4 難易度・山（インクリメンタル版）

- **時間経過**に応じて **コスト・需要**が上がる、**イベントウェーブ**で一時的に **3 択の頻度や報酬**が変わる、など **「盛り上がり」**を数値とイベントで作る（全方位スポーンの代替）。

## 4. UI / 情報設計（左ペイン想定）

| 領域 | 内容 |
|------|------|
| **左** | 主リソース、**生産量/sec**（または同等）、**施設リスト**、購入ボタン、**進行ログ**（短く）。 |
| **3 択** | コンパクトなカード（モーダルまたはインライン）。キーボードは **任意**（MVP はポインタのみでも可）。 |
| **右** | `CatMascot` + **ごきげん／報酬**の見た目。**生産効率や 3 択の内容と接続**できるとよい。 |
| **軽減** | `prefers-reduced-motion` で数字の派手なカウントを抑える。 |

---

## 5. データモデル・永続

- **状態**: リソース量、施設レベル、解放フラグ、経過時間、**猫／メタ用の少数パラメータ**。
- **保存**: 既存キーを **マイグレーション**で拡張（キャッチ時代の `highScore` 等は **無視 or 別名へ移す**方針を決める）。
- **VS 式選択**: `choiceId`、候補プール、シード（再現が要らなければ省略可）。

## 付録 A. 検討経過メモ（アクション寄り案）

以下は **主軸をインクリメンタルに寄せる前**に計画されていた内容の要約。**実装の一部に残っている可能性**があるが、**現行の公式方針では優先しない**。

- **運搬スプライトの 2D 移動**、**落下物のキャッチ**、**ライフ／コンボ**。
- **全方位スポーン・速度ベクトル（`vx` / `vy`）** で *VS* ライクな圧を出す案。
- これらを **完全に捨てる必要はない**（ミニゲーム化・別モード・イベント用素材として再利用可）。
