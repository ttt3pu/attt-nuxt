# おやつキャッチ Phase 2 — Vampire Survivors ライクなやり応え

**用語（本ファイル）**: **VS** は *Vampire Survivors*（移動＋自動系の攻撃・時間経過で難化・レベルアップでラン内ビルドが組めるローグライト）の略。**「対戦」や Visual Studio の意味ではない。** 以降 **VS** と書いた箇所はすべてこれを指す。

## 0. 位置づけ・前提

| 項目 | 内容 |
|------|------|
| **前身** | Phase 1 は [完了・アーカイブ](done/oyatsu-catch.md)。落下キャッチ、良／悪、ライフ、コンボ、難易度ランプ、`localStorage` 永続化、`CatMascot` 連動の土台がある。 |
| **目的** | いまのゲーム性（短時間でわかる操作・スコア・コンボ・猫）を **壊さず**、**Vampire Survivors（以下 VS）系の「ランが伸びる／ビルドが変わる」快感**を、このサイトのヒーロー内に収まる規模で取り込む。 |
| **やらないことの宣言** | 本作を **本物のスクロール系バレットヘル**や **フル画面専用の長時間ゲーム**にはしない。ポートフォリオの **脇役として遊べる深度**を目標にする。 |

**前提技術**（Phase 1 と同じく）: Nuxt / Vue、**DOM + CSS transform + `requestAnimationFrame`** を既定とする。粒子数やスプライトが増えてボトルネックになる場合のみ、**フィールド部分の Canvas 2D 化**などを段階的に検討する（専用ゲームエンジンは使わない）。

### 0.1 操作対象と `CatMascot`（設計として確定）

- **Vampire Survivors ライクな自由移動**（ゲームフィールド内の **上下左右**）を採用してよい。Phase 1 の「画面下端の受け皿を左右のみ動かす」実装は **出発点**とし、**運ぶ側のスプライト**を 2D に動かす形へ発展させる。
- **プレイヤーが操作するのは**、おやつを運ぶ側の見た目（**手・小さなキャラ・受け皿アイコン等** — アート・実装で選択）。**`CatMascot` 本体は操作しない。**
- **`CatMascot`** は **右列に固定**し、喜び／怒り／ごきげんメーターなど **感情・世界観の軸**として据える（グリグリ移動させない）。
- **ルールの再定義**: 受け皿1軸型で合意した **取り逃し・コンボ・経験値**などは、双方向移動＋落下／スポーンのモデルに合わせて **文言・判定を見直す**（セッションで決めた内容と突き合わせ、矛盾があれば更新する）。

## 1. VS から借りる「快感の骨格」（抽象）

VS が示す体験を、おやつキャッチに写像すると次のようになる。**字面は違ってよい**が、プレイヤーが感じるべきテンションの型は近づける。

1. **時間が経つほど追い詰められる**（生存プレッシャー）。逃げ切る・踏ん張る爽快感。
2. **成長の実感**が短いサイクルでループする（経験値 → レベルアップ → 何かが強くなる）。
3. **そのラン限りのビルド**（選択の連鎖で「この組み合わせで今回はこうなった」）。
4. **入力は単純**で、派手さや強さは **自動・派生（バフ・追加判定・演出）** で返す（猫・**運搬側**・落下物の関係で表現）。
5. **もう一度**の理由が、スコアだけでなく **ビルドの試行錯誤**と **記録更新**に分かれる。

Phase 1 の縦キャッチは **基礎**とし、上記を足しつつ、**セクション 0.1** のとおり **運搬側の自由移動**へ発展させてよい。

## 2. Phase 1 から継承するもの（変更しない前提）

次は **基本として維持**し、Phase 2 はその周辺に機能を足す。

- **導線**: トップヒーロー左のゲームエリア、`遊ぶ`／閉じる、プレイ中の `AtScroll` 制御など（`index.vue` のパターン）。
- **操作**: ポインタ／タッチで **運搬側（セクション 0.1）** の位置へ追従。Phase 1 は下端受け皿の **1 軸**、Phase 2 では **フィールド内 2 軸** を前提にできる。**猫は操作しない。**
- **アイテム種別**: 良いおやつ／悪いものの対立（世界観の核）。
- **ライフ制ゲームオーバー**: Phase 1 と同じく **ライフ 0** で終了を基本（時間切れのみの敗北は採りにくい）。
- **コンボ**: スコア快感の芯。Phase 2 では **XP やパワーと掛け合わせる**余地を残す。
- **永続データ**: `attt:oyatsu-catch` の **単一 JSON + `version`** 方針を継続し、フィールド追加で拡張する。
- **`CatMascot`**: 喜び／怒りリアクション、右側の雰囲気。**ごきげんメーターや「届ける」演出**は Phase 1 プラン（[done/oyatsu-catch.md](done/oyatsu-catch.md) の「4.4 右の猫を活かすゲーム性」）にあった未実装も、Phase 2 で **レベルアップ・パワーと一緒に設計**すると筋が良い。

## 3. Phase 2 で新設するコア（提案）

### 3.1 ラン内メタ：経験値とレベルアップ

- **経験値（XP）** と **プレイヤーレベル（ラン内）** を導入。一定 XP でレベルアップし、**一時停止または短いオーバーレイ**で **2〜3 択の「おやつパワー」**から 1 つ選ぶ（VS のレベルアップ選択の軽量版）。
- XP の獲得源（例・実装時に配分調整）:
  - 良いアイテムのキャッチ（基本）
  - コンボボーナス
  - 任意: 悪を **掴まずに回避し続けた時間** のごく小さなパッシブ（操作性に価値を出す）

### 3.2 おやつパワー（ビルド要素）

**そのプレイsession 内だけ**有効な強化。**スタック**や **シナジー**を少数から入れ、強すぎないよう **上限・排他**を型で表現できるとよい。

| 系統 | 例（仮） | 狙い |
|------|-----------|------|
| **受け皿** | 幅拡大、キャッチ判定の縦方向ゆとり、短時間だけ大型化 | 「動きの幅」を広げる |
| **磁石・吸引** | 良のみ / 一定範囲を受け皿へ寄せる | VS の磁石感を軽く |
| **落下・速度** | 全体スロー、良のみ落下遅延 | 窮地の逃げ道 |
| **悪への防御** | 1 回だけ悪キャッチを無効、ライフ減を 1 に軽減 | 緊張の緩衝 |
| **猫・ごきげん連動** | メーター上昇量アップ、MAX 時に短時間オート収集（良のみ）など | 右列 UI と統合 |
| **派手さ（軽量）** | 稀に悪を画面上部で弾く、など **低頻度オート** | 「自動で強くなる」体感（DOM/CSS のみでも可能な範囲で） |

具体ステータスは **プロト後のチューニング**でよいが、**選択肢ごとに役割が被りすぎない**よう ID 設計する。

### 3.3 難易度と「山」

- Phase 1 の `difficulty` に加え、**経過時間**と **ラン内レベル**に応じて:
  - スポーン間隔、落下速度、悪の出現率、サイズばらつきを調整。
- **イベントウェーブ**（例: 30〜45 秒ごとに短い高密度スポーン）で **手応えの山**を作る（VS の「画面が荒れる」感の極薄版）。

### 3.4 勝利条件（任意モード）

- デフォルトは **倒れるまで**（ライフ 0）でよい。
- 余裕があれば **「○分生存」クリア**や **ボスウェーブ撃破**など **短い目標モード**を別フラグで追加できるよう、ゲーム状態を `phase` 拡張しやすく書く。

## 5. データモデル・実装の伸ばし方

### 5.1 型と composable

- `OyatsuCatchPowerId`（列挙）、ラン内 `activePowers`（重ね掛けルール）、`level`, `xp`, `xpToNext` などを `useOyatsuCatchGame` または **`useOyatsuCatchRun` + 既存 composable** に分離するかは実装時に判断（ファイルが肥大化したら分割）。
- レベルアップ中は `playing` のサブ状態（例: `pausedForChoice`）で入力を止める。

### 5.2 永続（メタ）

Phase 1 の保存に、**後方互換のマージ**で足す候補:

- **統計**: 累計生存時間、最高レベル、最高スコア（既存ハイスコアと整合）。
- **アンロック**: パワーの「初出は出現プールに入れる」だけのメタでも可。重くしない。

`version` を上げ、既存キーは壊さない。
