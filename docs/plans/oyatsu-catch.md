# おやつキャッチ（Oyatsu Catch）実装プラン

## 0. 引き継ぎ・読み方

このファイルは **別セッション・別担当者が実装を続ける**ことを想定した設計メモである。

| 読む順序 | 内容 |
|----------|------|
| 1 | **§9.1** … 確定した仕様の一覧（チェック済み） |
| 2 | **§3〜4** … 画面遷移とゲームルール |
| 3 | **§6** … 描画・永続・コンポーネント方針 |
| 4 | **§9.2** … 未決・チューニング（プロト後に決めてよい） |
| 5 | **§8** … マイルストーン順に着手 |

**リポジトリ**: `attt-nuxt`（Nuxt トップページ `app/pages/index.vue` のヒーロー）。**前提**: 既存の `MoleculesCatMascot` を活かす。新規ゲーム専用エンジンは使わない。

**実装時に必ずクライアント側で扱うもの**: `localStorage`（§6.4）は SSR では使えない。読み書きは `import.meta.client` やマウント後に限定する。

**あえて書いていないこと（後続で決める）**: 良／悪アイテムを **取り逃したとき**の扱い（コンボ・ライフ・ペナルティの有無）。§9.2 に記載。§2 の「30〜90 秒」は **目安**（ライフ制のため長短あり）と解釈する。

## 1. 概要

トップページのヒーロー（`title-container`）付近で遊べる、短時間の縦スクロール系キャッチミニゲーム。既存の `MoleculesCatMascot` を世界観の中心に据え、スコア・コンボ・ランダム要素で「もう1回」を誘う。

## 2. ゴール

- **体験**: 初訪問者が数秒でルールを理解し、1プレイ 30〜90 秒程度で完結できる。
- **技術**: Nuxt / Vue の既存スタックのみで完結（追加の重いゲームエンジンは使わない）。
- **品質**: ヒーローの可読性・主要導線（スクロール、ロゴ）を損なわない。**モバイル（タッチ）と PC（マウス）の両方**で遊べる操作を前提にする。

## 3. 配置・導線

### プレイ開始（決定済み）

- **「遊ぶ」ボタンは常時表示**する（ヒーロー内の固定位置。スタイルは実装時に調整）。
- ボタン押下で **ヒーロー左半分**（`MoleculesSiteLogo` などのロゴ・タイトル系ブロック）を **非表示にし、その領域をゲームエリアに切り替える**。
- ゲーム終了（またはユーザーが戻る操作をしたタイミング）では **左半分を再表示**し、ヒーローの通常表示に戻す（戻り操作のラベルは「閉じる」「トップに戻る」等、実装時に決定）。

### その他

- `CatMascot` は **装飾として残し**（右側など）、プレイ中は視覚的に「下で受け止める猫」として連動させる余地を残す（§7）。
- z-index は `--z-cat` / `--z-title` と競合しないよう変数を整理する。
- 右半分のレイアウト（猫の見え方・`AtScroll` との重なり）はブレークポイントごとに要確認（特に `width <= 768px` の縦積み時）。

## 4. コアルール（MVP）

1. **プレイフィールド**: 縦長の矩形。上からアイテムが落下する。
2. **プレイヤー**: 画面下部の **受け皿（バスケット）** を左右に移動する。入力の詳細は **§4.3**。
3. **良いアイテム**: キャッチで **スコア加算**。連続キャッチで **コンボ倍率**（上限あり）。
4. **悪いアイテム**: 登場し、**キャッチするとペナルティ**（主に **ライフ減少**。種類によって減り方を変えてもよい）。コンボリセットや減点との **併用**は実装時に調整可。
5. **ライフ**: 初期値は実装時にチューニング（§9.2）。**ライフが 0** で **ゲームオーバー**。残りライフを UI で常時表示する。
6. **ゲームオーバー後**: スコア・ハイスコア表示、リトライやヒーローへの戻り（§3）。

※ **時間切れのみ**でのゲームオーバーは MVP では採用しない（必要なら将来モードとして検討）。

### 4.1 ランダム要素（MVP 範囲）

- 落下物の **種類・横位置・間隔** をランダム化（シード不要。`Math.random` で可）。
- 一定スコア or 時間経過で **出現テーブル** を変化（例: 悪いアイテム比率がわずかに上がる）。

### 4.2 成長・リプレイ性（MVP 以降で拡張）

- **ラン内**: コンボ、一時的な出現バフ（レアおやつ）など。
- **メタ（永続・決定済み）**: 少なくとも次を保存する。
  - **ハイスコア**
  - **累計プレイ回数**
  - **実績**（解除フラグ・進捗など。表現は実装時に決定）
  - その他（**スキン解放**など）も同じ仕組みに載せられるようにする。
- 今後フィールドを増やす前提のため、保存形式は **§6.4** のとおり **拡張しやすい形**にまとめる（キーを乱立させない）。

### 4.3 操作（決定済み）

- **モバイル**: **タッチ**でプレイ可能（フィールド上の指の位置・ドラッグ等に受け皿を追従させる方式は実装時に決定）。
- **PC**: **マウス操作を前提**（クリック／ポインタ位置に受け皿を合わせる想定。`pointer` イベントでモバイルとロジックを共通化してよい）。
- **キーボード**（矢印キー等）は MVP では **必須にしない**。必要なら後から **アクセシビリティ用オプション**として追加（§9.2）。

### 4.4 右の猫を活かすゲーム性（決定済み・次期実装）

レイアウト上 **左がプレイフィールド・右が `CatMascot`** であることを活かし、次の 3 点を **セットで** 入れる（実装順は §8）。

#### （1）「猫に届ける」演出

- **良いアイテム**をキャッチしたタイミングで、視覚的に **右の猫へおやつが届く**イメージを付与する。
- 実装案: キャッチ地点から **短い移動／吸い込み風**の CSS アニメ（一時要素 or フィールド内オーバーレイ）。`CatMascot` 本体の座標に厳密に追従させなくても、**右方向・猫の高さ付近**をターゲットにすれば十分。
- **`prefers-reduced-motion: reduce`** ではアニメを省略 or フェードのみにする。

#### （2）猫のリアクション（プレイ状態と連動）

- **良キャッチ・コンボ上昇**に応じて `CatMascot` に **喜び**用の CSS クラスを短時間付与（既存の顔まわりパーツを活かす。§7 P1 と一体で設計）。
- **悪キャッチ・ライフ減**に応じて **しょんぼり／ビクッ**等の別クラス。
- 連続イベントでの **クラス打ちっぱなし**を避けるため、短いクールダウン or 優先度付けを検討する。

#### （4）ごきげん／満腹メーター（右側 UI）

- ゲーム中、**右列（猫の近傍）**に **メーター UI** を置く（左の HUD と役割分担: 左＝スコア・ライブ、右＝猫の機嫌）。
- **良いおやつ**でメーター上昇。**悪いおやつ**やライフ減に連動した **減少／ペナルティ**の有無は実装時に決定（§9.2）。
- メーターが **一定以上（MAX）** に達したら **短時間バフ**を発動（例: 受け皿幅アップ、落下速度低下、スポーン間隔の緩和）。**バフの具体値・持続時間**は §9.2。
- MAX 時は（2）の喜び表現と組み合わせてもよい。

## 5. UI / アクセシビリティ

- **操作**: **タッチ（モバイル）とマウス（PC）** を主とする（§4.3）。`prefers-reduced-motion` ではアニメを弱める or オフ。
- **視認性**: コントラスト、タップ領域の最小サイズを確保。**残りライフ**が一目で分かる表示（アイコン・ハート等は実装時に決定）。
- **ごきげんメーター**（§4.4（4））は **右列**に置き、猫と視覚的にまとまるレイアウトにする（モバイル縦積み時は折りたたみ or 猫直上など要検討）。
- **スクリーンリーダー**: ゲーム開始 / 終了時に `aria-live` でスコアを通知するか検討（実装時に詳細化）。

## 6. 実装方針（技術）

### 6.1 描画（決定済み）

**DOM + CSS transform** で実装する。位置更新は **`requestAnimationFrame`**（または `@vueuse/core` の `useRafFn` 等）。

| 方式 | メリット | デメリット |
|------|----------|------------|
| **DOM + CSS transform**（採用） | Vue テンプレートと相性が良い、Cat との重ねが容易 | 大量スプライトには不向き（本ゲーム規模なら十分なことが多い） |
| Canvas 2D | 描画の自由度 | 状態と描画の二重管理 |

パフォーマンス問題が出た場合のみ **Canvas への切り替え**を検討する。

### 6.2 コンポーネント分割（案）

- `OrganismsOyatsuCatch`（または `Molecules`）: フィールド・入力・状態マシン。
- `useOyatsuCatchGame.ts`（composable）: スコア、コンボ、**ライフ**、スポーン、衝突判定、ゲームループ。§4.4 追加後は **ごきげんメーター・バフ状態**もここまたは別 composable に分離。
- 定数・型: `types/oyatsu-catch.ts` または composable 内に閉じる。

### 6.3 状態

- `idle` → `playing` → `gameover` → `idle`。
- 外部ストア必須ではない。必要ならプレイ結果だけ emit または軽い store。

### 6.4 永続データ（決定済み）

- **保存先**: `localStorage`（キーは **1 本に集約**し、プロジェクト内で名前の衝突がないようプレフィックスを付ける。例: `attt:oyatsu-catch`）。
- **形式**: **JSON オブジェクト 1 個**としてシリアライズする。フィールドを後から追加しても読み込み側で **デフォルト値とマージ**できるようにする。
- **スキーマバージョン**: オブジェクトに **`version`（整数）** を含め、将来の構造変更時は **読み込み時にマイグレーション**（旧 `version` → 新オブジェクト）を書けるようにする。
- **型**: TypeScript で **型＋パース**（`zod` 等の利用は任意。未導入なら手書きでも可）し、未知キーは捨てずに **余剰プロパティを保持**するかどうかは実装時に決める（保持すると第三者拡張に強い）。
- **実績**: ID をキーにした **マップ**、または ID の **配列** など、一覧表示・クエリしやすい形を採用する（詳細は §9.2 で詰める）。

## 7. CatMascot との連動（フェーズ分け）

| フェーズ | 内容 |
|----------|------|
| P0 | ゲームエリアは `title-container__cat` と **レイアウト上隣接**するのみ（現状）。 |
| P1 | **§4.4（2）**: 良／悪キャッチ・コンボに応じた **CSS クラス**（喜び・しょんぼり等）を短時間付与。 |
| P1b | **§4.4（1）**: 良キャッチ時の **届ける／吸い込み**アニメ（DOM + CSS）。 |
| P1c | **§4.4（4）**: **ごきげんメーター**と MAX 時 **バフ**（ゲームロジック＋右側 UI）。 |
| P2 | マスコットの **口・耳** を別レイヤーにして細かく反応（任意・工数大なら後回し）。 |

（1）（2）（4）は **§4.4 で採用決定済み**。実装は P1 → P1b → P1c の順がわかりやすいが、並行も可。

## 8. マイルストーン

1. **M1**: プラン確定（本ドキュメント）＋ワイヤー相当の画面構成メモ。
2. **M2**: プロトタイプ（移動・1 種類の落下・スコア・ゲームオーバー）。
3. **M3**: 良/悪アイテム、コンボ、難易度曲線、モバイル操作の調整。
4. **M4**: 永続データ（ハイスコア・累計・実績など §6.4）、ポリッシュ、アクセシビリティ、`prefers-reduced-motion`。
5. **M5**: **§4.4** に基づく **猫連動**（届ける演出・リアクション・ごきげんメーター／バフ）。詳細は §7。
6. **M6（任意）**: スキン、サウンド、§7 P2。
